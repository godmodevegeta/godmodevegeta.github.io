1. Product Requirements (PRD)

1.1 Vision

Lifeline takes the ultra-sophisticated, high-speed incident response architecture used by FAANG engineering teams and repurposes it to solve the most critical problem in the non-profit sector: resource dispatch. It proves that agentic workflows don't just save developer hours — they save lives.

By autonomously matching urgent human needs (reported in Slack) with real-time inventory, volunteer locations, and shelter capacities (via MCP), Lifeline transforms the "NGO Context Tax" from a 45-minute administrative nightmare into a 4-second dispatch decision.

1.2 Problem

The NGO Context Tax

Social impact organizations run their operations on Slack. But their data lives in a chaotic web of Google Sheets, Airtable databases, and legacy CRMs. When a frontline volunteer posts in #intake:

"Client needs emergency housing tonight. Has a service dog, requires wheelchair access."

The dispatcher spends 45 minutes frantically checking four different spreadsheets and making phone calls. Meanwhile, the vulnerable person waits.

Time Is Literally Trauma

Every minute of administrative friction is a minute of uncertainty for someone in crisis. Current solutions are either:

  • Expensive and rigid — legacy NGO tech requires six-month procurement cycles
  • Technically weak — off-the-shelf Slack bots can't query complex, multi-variable database constraints dynamically
  • Human-bottlenecked — dispatchers burn out fighting databases instead of helping humans

Volunteer Burnout

The number one cause of churn in social impact is administrative friction. Volunteers want to help humans, not fight databases. When a volunteer spends 45 minutes on hold or cross-referencing spreadsheets for a single placement, they don't return next week.

1.3 The Idea

Lifeline is a highly tuned agent built for the Slack Agent for Good track. It leverages the "Holy Trinity" of hackathon technologies — RTS API, MCP Server Integration, and Slack AI — to execute a flawless, life-saving workflow.

The Frontline Workflow

StepTechnologyWhat Happens
1. Trigger RTS API A volunteer drops a complex constraint request into #intake. Lifeline is summoned via a message action. It ingests the exact parameters of the crisis — wheelchair-accessible, pet-friendly, immediate capacity, proximity to Zone A — plus ambient workspace context from channels like #logistics-alerts.
2. The Lifeline Matrix MCP (Parallel) Lifeline executes parallel, read-only queries across the NGO's disjointed systems:
  • Shelter Roster MCP (Airtable): Filter for Capacity > 0, Wheelchair = True, Pet = True
  • Transport Roster MCP (Google Sheets): Find available volunteer drivers with vans in Zone A
3. Ontology Validation Ontology MCP Before presenting any match, Lifeline validates physical compatibility through a deterministic constraint engine. For example: a motorized wheelchair (350 lbs) does not fit a standard passenger van lift (300 lbs). The ontology rejects the match before the LLM ever sees it. No hallucination. No override.
4. Synthesis Slack AI Lifeline's LLM synthesizes the validated database hits into natural language: "Match found: Hope Shelter has 1 bed. Volunteer Dave is 5 miles away with a heavy-duty van."
5. Dispatch Card Block Kit A mobile-optimized Action Card drops into the thread: exact match, confidence level, ambient alerts (e.g., road closures), and an actionable button: "Preview Dispatch"
6. Human Approval Slack User Token The dispatcher clicks Preview Dispatch, reviews the draft messages and resource lock in a modal, then clicks Confirm & Send. The AI is an exoskeleton for the volunteer, not an unsupervised decision-maker.

1.5 Impact & Scale

Immediate Impact (0–6 months)

  • Dispatch velocity: 10x increase in cases handled per volunteer hour
  • Volunteer retention: 40% reduction in administrative-friction-related churn
  • Error elimination: Zero missed constraint matches through deterministic ontology validation
  • Cost accessibility: Mid-sized shelters ($0 software budget) gain FAANG-grade dispatch infrastructure

Medium-Term Impact (6–18 months)

  • The No-Code MCP Flywheel: By open-sourcing generic Google Sheets and Airtable MCP servers, Lifeline gives NGOs the pipes to connect any data to AI — not just shelter rosters, but food bank inventory, medical supply chains, evacuation route capacities
  • Cross-NGO coordination: Regional networks of shelters share real-time capacity via MCP, preventing overflow at one location while beds sit empty 10 miles away
  • Disaster response: During hurricanes, wildfires, or refugee crises, Lifeline scales from shelter dispatch to multi-agency resource coordination

Long-Term Impact (2–5 years)

  • Standardizing NGO ChatOps: Lifeline becomes the default operational OS for shelters, food banks, crisis centers, and disaster relief teams worldwide
  • From reactive to predictive: Historical dispatch patterns + weather APIs + census data → proactive resource pre-positioning before crises peak
  • Global health applications: The constraint-matching architecture ports to hospital bed management, vaccine distribution, and emergency medicine triage

Second-Order Effects

By reducing administrative friction, NGOs see higher volunteer retention rates, indirectly multiplying their workforce. A 40% retention improvement doesn't just save recruitment costs — it means 40% more human hours available for direct service.

2.3 Data Flows

Flow A: Successful Match (Golden Path)

Volunteer posts in #intake:
  "@Lifeline Need emergency housing for a family of 3. Wheelchair, service dog. Zone B."

  → User clicks "Summon Lifeline" message shortcut
    → ack()
    → Spinner: "Gathering context..."

    [STEP 1: Ambient Context Sweep]
      → assistant.search.context(
          channel_ids: [#logistics-alerts, #shelter-ops],
          query: "Zone B AND (closure OR outage OR full)",
          timeframe: "last_2_hours"
        )
      → Result: "⚠️ Mission St (Zone B) closed due to water main break"

    → Spinner: "Querying resources..."

    [STEP 2: Parallel MCP Fetch]
      → Promise.all([
          mcp.airtable.query_shelters(zone: "B", min_capacity: 3, wheelchair: true, pets: true),
          mcp.sheets.query_transport(zone: "B", vehicle_class: ["Van", "SUV"], available: true)
        ])
      → Shelter: St. Jude's Center (4 beds, 400 4th Ave — NOT on Mission St)
      → Transport: Dave (Passenger Van, available, English/Spanish)

    → Spinner: "Validating constraints..."

    [STEP 3: Ontology Validation]
      → mcp.ontology.evaluate_compatibility(
          need: { motorized_wheelchair: true, service_dog: true, family_size: 3 },
          vehicle: { type: "passenger_van", lift_capacity_lbs: 300, passenger_capacity: 6 }
        )
      → Result: { compatible: true }
      → (Terminal log visible to demo audience: validation passed)

    → Spinner: "Building dispatch..."

    [STEP 4: Synthesis + Block Kit Render]
      → Slack AI synthesizes validated match into natural language
      → Block Kit Action Card drops in thread:

        🚨 LIFELINE DISPATCH MATCH

        Client Need: Family of 3, Wheelchair, 1 Service Dog (Zone B)

        🏠 Shelter Match: St. Jude's Center
           • 400 4th Ave, Zone B (off Mission St — accessible)
           • 4 beds available, Wheelchair accessible, Pets allowed

        🚐 Transport Match: Dave
           • Passenger Van, lift-equipped
           • ETA 12 min, English + Spanish

        ⚠️ Ambient Alert: Mission St closed (water main break).
           Dave must route via 4th Ave.

        [ 📋 Preview Dispatch ]  [ 🔄 Re-run Search ]

    → Spinner cleared. Card posted.
    → logToChannel(#lifeline-logs, full audit)

Flow B: Ontology Rejection (Constraint Failure)

Same trigger, but client has motorized wheelchair (350 lbs)

  → Shelter: St. Jude's Center (passes)
  → Transport: Dave (Passenger Van, lift capacity 300 lbs)

  → Ontology validation:
     mcp.ontology.evaluate_compatibility(
       need: { motorized_wheelchair_weight_lbs: 350 },
       vehicle: { lift_capacity_lbs: 300 }
     )
  → Result: { compatible: false, reason: "Lift capacity exceeded" }

  → Lifeline does NOT surface Dave. Instead:
     • Re-queries Sheets for heavy_duty_van or wheelchair_accessible_van
     • Finds Lisa (Heavy-Duty Van, lift capacity 500 lbs)
     → Result: { compatible: true }

  → Block Kit shows:
     🚐 Transport Match: Lisa
        • Heavy-Duty Van, 500 lb lift
        • ETA 11 min
        • (Dave rejected: lift capacity insufficient — see audit log)

  → Terminal log (demo-visible):
     [ONTOLOGY] Rejected vehicle vol_01 (Dave)
     [ONTOLOGY] Reason: motorized_wheelchair (350 lbs) > lift_capacity (300 lbs)
     [ONTOLOGY] Selected vehicle vol_06 (Lisa)
     [ONTOLOGY] Reason: motorized_wheelchair (350 lbs) <= lift_capacity (500 lbs)

Flow C: Autonomous Constraint Relaxation (No Exact Match)

Trigger: "Need housing for family of 3, wheelchair, service dog, Zone B"

  → Step 1: Strict query to Airtable MCP
     filter: { zone: "B", min_capacity: 3, wheelchair: true, pets: true }
     → Result: [] (zero shelters match all constraints)

  → Step 2: Agent autonomously relaxes lowest-priority constraint
     → System prompt: "If strict query returns empty, drop the lowest-priority
        constraint and re-query. Priority order: zone > wheelchair > pets > capacity"
     → Relaxed query: { zone: "B", min_capacity: 3, wheelchair: true, pets: ANY }
     → Result: [St. Jude's Center (pets allowed), Hope Shelter (pets allowed)]
     → Both pass. No further relaxation needed.

  → Step 3: If relaxed query still returns empty
     → Drop next constraint: wheelchair = ANY
     → Result: [Tenderloin Housing Clinic (no wheelchair, no pets)]
     → Still fails ontology (wheelchair required). Continue relaxation...

  → Step 4: Final fallback — present partial matches to human
     → Block Kit shows:
        ⚠️ No exact match found. Closest options:
        • St. Jude's Center — meets 3/4 constraints (missing: ???)
        • Hope Shelter — meets 3/4 constraints (missing: ???)
        • [ 🔍 Expand to Zone A+C ]  [ 📞 Escalate to Supervisor ]

  → Human dispatcher chooses next action.

Dispatcher clicks "📋 Preview Dispatch"

  → views.open(modal: {

2.4 Security & Trust Model Fix 1

PrincipleImplementationWhy It Matters for NGOs
Read/Write Separation MCP servers are read-only for all inventory queries. lock_resource executes only after human confirmation, with the dispatcher's Slack user_id passed as an explicit parameter in the MCP tool call. The MCP server logs this user_id in its audit trail and verifies write permissions. The AI researches. The human decides. Every write is attributable to a specific human. No anonymous bot actions.
Deterministic Ontology Validation Physical constraints are validated by a standalone symbolic engine before any LLM synthesis. The LLM cannot override a compatible: false result. Eliminates hallucination risk for life-critical resource allocation. If the ontology says no, the AI cannot invent a match.
Two-Button Approval "Preview Dispatch" → modal review → "Confirm & Send." When the dispatcher clicks confirm, the Bot extracts user.id from the Slack block_actions payload and passes it as initiated_by_user_id to the MCP lock_resource call. No single-click execution. Dispatcher sees exactly what will be locked and sent before confirming. Prevents accidental dispatches on mobile devices. Respects the dispatcher's judgment. Every action is attributable to a specific human without requiring OAuth2 user token management.
100% Audit Logging Every invocation appends structured data to #lifeline-logs: RTS payload, MCP queries, ontology validation results, LLM prompt, human approvals, token identity. NGOs operate under grant compliance and donor transparency requirements. Audit logs are non-negotiable.
Least-Privilege Scopes Bot: chat:write, channels:read, commands, assistant:write. User: chat:write (for approved dispatches only). Slack Marketplace non-negotiable. Minimizes blast radius if credentials are compromised.

2.5 Error Handling & Fallbacks Fix 3

Failure ModeFallback BehaviorUser Experience
Airtable MCP timeout (>3s) Return partial data from Sheets MCP. Flag missing shelter data. Suggest manual check. "⚠️ Shelter roster temporarily unavailable. Transport match found. Please verify capacity manually."
Ontology MCP timeout or crash Skip validation and flag: "Manual verification recommended." Do not block the flow. "⚠️ Constraint validator offline. Match presented without physical validation. Please verify manually."
No exact matches (strict query) Autonomous constraint relaxation: Agent drops lowest-priority constraint and re-queries MCP. Only if relaxed query also returns [] does it fallback to partial match UI. "Expanding search to include shelters without pet policy..." → "Found 2 matches with relaxed constraints." OR "No matches found. Closest partial matches below."
lock_resource fails (race condition) Notify dispatcher: "Beds no longer available. Re-running search." Auto-retry with updated capacity. "⚠️ St. Jude's beds were just claimed. Re-searching with live data..."
Dispatcher lacks User Token scope Show modal: "Connect your account to send dispatches." OAuth re-prompt. One-click re-auth. No admin intervention required.

2.6 MCP Server Specifications

Airtable MCP (Shelter Roster)

Source: Official Airtable MCP server (https://mcp.airtable.com/mcp) with OAuth and PAT authentication.

Tools Exposed:
• query_shelters(zone, min_capacity, wheelchair, pets, family_units)
• lock_resource(shelter_id, resource_type, quantity, initiated_by_user_id)
• unlock_resource(shelter_id, resource_type, quantity, initiated_by_user_id)

Query Example:
{
  "table": "Shelters",
  "filter": {
    "Zone": "B",
    "Capacity Remaining": { ">": 0 },
    "Wheelchair Accessible": true,
    "Pet Friendly": true,
    "Family Units": true
  },
  "sort": [{ "field": "Capacity Remaining", "direction": "desc" }],
  "max_records": 5
}

Lock Example:
{
  "shelter_id": "rec456",
  "resource_type": "bed",
  "quantity": 3,
  "initiated_by_user_id": "U1234567890"
}

Google Sheets MCP (Transport Roster)

Source: Community-standard Google Sheets MCP server with service account authentication.

Tools Exposed:
• query_transport(zone, vehicle_class, wheelchair, pets, available)
• lock_resource(volunteer_id, resource_type, status, initiated_by_user_id)
• unlock_resource(volunteer_id, resource_type, status, initiated_by_user_id)

Query Example:
{
  "spreadsheet_id": "1a2b3c4d...",
  "sheet": "Volunteers",
  "filter": {
    "Zone": "B",
    "Available Now": true,
    "Vehicle Type": ["Van", "SUV", "Heavy-Duty Van"],
    "Wheelchair Accessible": true
  },
  "sort": [{ "field": "Distance Miles", "direction": "asc" }],
  "max_records": 3
}

Lock Example:
{
  "volunteer_id": "vol_01",
  "resource_type": "status",
  "status": "On Transport Run",
  "initiated_by_user_id": "U1234567890"
}

Ontology MCP (Custom FastMCP Server) Fix 4

Source: Standalone Python/Node HTTP service built with FastMCP. Not a third-party integration — this is Lifeline's deterministic constraint engine.

Hot-Reload Architecture: The ontology schema is loaded from ONTOLOGY_CONFIG_PATH (environment variable pointing to a JSON file or a Slack config channel message). For the hackathon, the schema is hardcoded. In production, NGO admins can update constraints via a simple Slack command (e.g., /lifeline-config add-vehicle bariatric_wheelchair 600lbs) or by editing the JSON in #lifeline-admin. The MCP server hot-reloads the schema without restart.

Tools Exposed:
• evaluate_compatibility(need_profile, vehicle_profile)
• validate_shelter_capacity(shelter_profile, need_profile)
• list_constraint_violations(match_candidate)
• reload_ontology()  // hot-reload from config source

Evaluate Compatibility Example:
{
  "need_profile": {
    "mobility_aids": ["motorized_wheelchair"],
    "mobility_aid_weight_lbs": 350,
    "service_animals": 1,
    "animal_type": "dog",
    "family_size": 3,
    "family_composition": { "adults": 1, "children": 2 }
  },
  "vehicle_profile": {
    "type": "passenger_van",
    "lift_capacity_lbs": 300,
    "passenger_capacity": 6,
    "wheelchair_slots": 1,
    "pet_capacity": 2,
    "cargo_cu_ft": 80
  }
}

Response:
{
  "compatible": false,
  "violations": [
    {
      "constraint": "lift_capacity",
      "need_value": 350,
      "vehicle_capacity": 300,
      "severity": "blocking",
      "message": "Motorized wheelchair (350 lbs) exceeds lift capacity (300 lbs)"
    }
  ],
  "recommendation": "Query for heavy_duty_van or wheelchair_accessible_van with lift_capacity >= 350"
}

2.9 Spinner Verbs (No Streaming Task Cards)

Lifeline uses simple status text updates instead of native streaming task cards:

1. setStatus("Gathering context...")        // RTS API sweep
2. setStatus("Querying resources...")         // Parallel MCP fetch
3. setStatus("Validating constraints...")     // Ontology MCP
4. setStatus("Building dispatch...")          // Slack AI synthesis + Block Kit
5. setStatus("")                              // Clear spinner, card posted

Each status update replaces the previous one in the thread. No incremental chunks. No task cards. Clean, minimal, mobile-friendly.
